What is ARP?

THE PROTOCOL THEY FORGOT TO SECURE

ARP — Address Resolution Protocol — is how devices on a local network translate IP addresses into MAC addresses. When your laptop wants to talk to 192.168.1.1, it broadcasts "who has 192.168.1.1?" across the entire LAN. The gateway responds with its MAC address and the conversation begins.

THE PROBLEM

ARP was designed in 1982 for trusted networks. It has no authentication. Any device can send an ARP reply claiming to own any IP. Devices accept these replies without question and update their routing tables immediately. This is the root cause of every attack KTOx demonstrates. 

# Normal ARP exchange:
Device A  →  BROADCAST:  "Who has 192.168.1.1?"
Gateway   →  Device A:   "192.168.1.1 is at AA:BB:CC:DD:EE:FF"

# Poisoned ARP exchange:
Attacker  →  Device A:   "192.168.1.1 is at 11:22:33:44:55:66"  (lie)
Attacker  →  Gateway:    "192.168.1.50 is at 11:22:33:44:55:66" (lie)
# Now all traffic flows through the attacker

// attack surface

The Attack Modules

MODULE 01
THREAT: HIGH
ARP DENIAL OF SERVICE
Sends a continuous stream of forged ARP replies to a target, telling it that the gateway's MAC address is the attacker's MAC. The target routes all traffic toward the attacker, who does not forward it — effectively cutting the target off from the network.
Modes: KICK ONE · KICK SOME · KICK ALL
Re-ARPs on stop to restore connectivity.
MODULE 02
THREAT: CRITICAL
ARP CACHE POISONER — MITM
Bidirectional ARP poison. The attacker tells the target "I am the gateway" and simultaneously tells the gateway "I am the target." All traffic between them flows through the attacker's machine, enabling passive interception of credentials, sessions, and unencrypted data.
Requires IP forwarding enabled to avoid dropping packets:
echo 1 > /proc/sys/net/ipv4/ip_forward
MODULE 03
THREAT: HIGH
ARP FLOOD — DoS
Floods a target with thousands of ARP replies per second, each containing a randomly generated fake IP and MAC pair. This saturates the device's ARP cache table. When the table is full, the device cannot learn legitimate mappings and network communication degrades or halts entirely.
Configurable packets-per-second rate.
Particularly effective against embedded routers and IoT devices.
MODULE 04
THREAT: MEDIUM
GRATUITOUS ARP BROADCAST
Broadcasts an unsolicited ARP reply to the entire network segment claiming ownership of any specified IP address. Every listening host on the LAN updates its ARP cache to associate that IP with the attacker's MAC. Useful for IP takeover, HSRP/VRRP bypass, and cache poisoning validation.
Destination: ff:ff:ff:ff:ff:ff (broadcast)
Configurable repeat count and interval.
MODULE 05
ROLE: PASSIVE
ARP WATCH — CONFLICT MONITOR
Passively sniffs all ARP traffic on the local interface. Builds a live IP→MAC mapping table and fires an alert whenever it detects a conflict — an IP address being claimed by a different MAC than previously recorded. Use this to detect if someone is already poisoning your network.
Passive — sends no packets.
All conflicts logged to JSON session file.
MODULE 06
ROLE: RECON
TARGET RECON + PORT SCAN
Resolves MAC address, hardware vendor (from local OUI table), and hostname for any target on the LAN. Follows up with a fast nmap port scan to enumerate open services. All results written to the session loot file.
Vendor lookup is local — no API, no network required.
Port scan: nmap -F -T4

How to Protect Yourself

ARP attacks are possible because the protocol has no built-in authentication. The defenses below range from passive detection to active prevention. Layer them — no single measure is sufficient on its own.

🔒
STATIC ARP ENTRIES
Manually map critical IPs (especially your gateway) to their MAC addresses. Static entries cannot be overwritten by ARP replies, making poisoning impossible for those hosts.
# Linux — add static ARP entry
sudo arp -s 192.168.1.1 AA:BB:CC:DD:EE:FF
🛡
DYNAMIC ARP INSPECTION (DAI)
Enterprise switches support DAI — they validate ARP packets against a DHCP snooping binding table and drop any that don't match. Stops ARP spoofing at the switch level before it reaches hosts.
# Cisco IOS
ip arp inspection vlan 1
ip dhcp snooping vlan 1
👁
ARP WATCH / MONITORING
Run passive ARP monitoring tools that alert on IP→MAC mapping changes. KTOx's own ARP Watch module detects conflicts in real time. Tools like arpwatch can email alerts when suspicious activity is detected.
sudo apt install arpwatch
sudo arpwatch -i eth0
🔐
ENCRYPT EVERYTHING
MITM attacks only expose data that's unencrypted. Use TLS/HTTPS for all web traffic, SSH instead of Telnet, and VPNs for sensitive communications. Even if an attacker intercepts your traffic, encryption makes it useless.
# Force HTTPS — install HTTPS Everywhere
# Use Wireguard or OpenVPN on untrusted networks
📡
VPN ON UNTRUSTED NETWORKS
On networks you don't control — coffee shops, hotels, shared offices — assume ARP poisoning is possible. A VPN tunnels all traffic through an encrypted channel, bypassing any local interception entirely.
# Wireguard — fast, modern, audited
sudo wg-quick up wg0
🧱
NETWORK SEGMENTATION
ARP attacks are limited to the local broadcast domain. VLANs segment your network so that an attacker on one VLAN cannot reach hosts on another. Isolate IoT devices, guest networks, and critical infrastructure into separate VLANs.
# Separate VLANs for IoT / Guest / Infra
# ARP cannot cross VLAN boundaries
⚡
PORT SECURITY
Configure managed switches to limit the number of MAC addresses per port. This prevents ARP flood attacks from registering thousands of fake MACs and also limits rogue device connections to your network.
# Cisco — limit MACs per port
switchport port-security maximum 2
switchport port-security violation restrict
🔍
XDR / IDS MONITORING
Intrusion detection systems like Snort, Suricata, or Zeek have rules for detecting ARP anomalies — high ARP rates, MAC address changes, and gratuitous ARP storms. Set up alerts to catch attacks in progress.
# Suricata ARP spoof detection
sudo suricata -c /etc/suricata/suricata.yaml -i eth0
