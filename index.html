// background

What is ARP?

THE PROTOCOL THEY FORGOT TO SECURE

ARP — Address Resolution Protocol — is how devices on a local network translate IP addresses into MAC addresses. When your laptop wants to talk to 192.168.1.1, it broadcasts "who has 192.168.1.1?" across the entire LAN. The gateway responds with its MAC address and the conversation begins.

THE PROBLEM

ARP was designed in 1982 for trusted networks. It has no authentication. Any device can send an ARP reply claiming to own any IP. Devices accept these replies without question and update their routing tables immediately. This is the root cause of every attack KTOx demonstrates. 

# Normal ARP exchange:
Device A  →  BROADCAST:  "Who has 192.168.1.1?"
Gateway   →  Device A:   "192.168.1.1 is at AA:BB:CC:DD:EE:FF"

# Poisoned ARP exchange:
Attacker  →  Device A:   "192.168.1.1 is at 11:22:33:44:55:66"  (lie)
Attacker  →  Gateway:    "192.168.1.50 is at 11:22:33:44:55:66" (lie)
# Now all traffic flows through the attacker

// offensive modules

Attack Modules

KTOx v5.0 includes 9 offensive modules covering every major ARP attack vector. All attacks are configurable with rate control and packet caps, and restore ARP tables cleanly on exit.

MODULE 01 · 02 · 03
THREAT: HIGH
ARP DENIAL OF SERVICE
Sends forged ARP replies to targets telling them the gateway's MAC is the attacker's. Traffic is routed toward the attacker and dropped — cutting the target off the network. Three modes: single target, multiple selected targets, or every non-gateway host at once.
Modes: KICK ONE · KICK SOME · KICK ALL
Configurable packets/min + total packet cap. Re-ARPs on stop.
MODULE 04
THREAT: CRITICAL
ARP CACHE POISONER — MITM
Bidirectional ARP poison. Tells the target "I am the gateway" and simultaneously tells the gateway "I am the target." All traffic between them flows through the attacker's machine — enabling passive interception of credentials, sessions, and unencrypted data in real time.
Requires IP forwarding to avoid dropping packets:
echo 1 > /proc/sys/net/ipv4/ip_forward
MODULE 05
THREAT: HIGH
ARP FLOOD — SINGLE TARGET DoS
Hammers a single target with thousands of ARP replies per second, each with a randomly generated fake IP and MAC pair. Saturates the device's ARP cache table. When full, the device cannot learn legitimate mappings and communication halts. Particularly effective against embedded routers and IoT devices.
Configurable packets-per-second rate + packet cap.
Target specified by IP address or host index.
MODULE 06
THREAT: HIGH
ARP REPLY STORM — BROADCAST DoS
Unlike the targeted flood, the ARP Reply Storm sends forged replies to ff:ff:ff:ff:ff:ff — the broadcast address. Every host on the segment receives and processes these fake entries simultaneously, degrading the entire LAN rather than a single device.
Destination: ff:ff:ff:ff:ff:ff (entire segment)
Configurable pps rate and packet cap.
MODULE 07
THREAT: MEDIUM
GRATUITOUS ARP BROADCAST
Sends unsolicited ARP replies claiming ownership of any specified IP address. Every listening host on the segment updates its ARP cache to associate that IP with the attacker's MAC. Useful for IP takeover tests, HSRP/VRRP bypass, and cache poisoning validation.
Configurable claim IP, source MAC, repeat count, interval.
Destination: ff:ff:ff:ff:ff:ff (broadcast).
MODULE 08
THREAT: HIGH
GATEWAY DoS — ROUTER TAKEDOWN
Instead of attacking clients, this module floods the gateway itself with thousands of fake ARP entries. The router's ARP table fills with bogus mappings and it can no longer route traffic for anyone on the LAN — taking down connectivity for the entire network simultaneously.
Target: the default gateway IP/MAC.
Configurable pps rate and packet cap.
MODULE 09
THREAT: CRITICAL
ARP CAGE — FULL LAN ISOLATION
The most complete isolation attack. Rather than just poisoning the gateway, the ARP Cage poisons the target's view of every other host on the subnet simultaneously — replacing every peer's real MAC with the attacker's. The target cannot communicate with anyone on the LAN, not just the gateway.
Poisons target → all N peers in parallel.
Full re-ARP on all peers on exit to restore connectivity.

// recon modules

Reconnaissance

MODULE A
ROLE: RECON
ARP REQUEST SCAN — STEALTH DISCOVERY
Sends ARP WHO-HAS requests across the entire subnet and collects replies. Faster than nmap's host discovery and largely invisible to IDS/IPS rules — most monitoring tools don't flag ARP requests as suspicious. Returns a full host table with IP, MAC, and vendor resolved from the local OUI database.
No packets sent beyond normal ARP.
Vendor resolved locally — no API, instant.
MODULE B
ROLE: RECON
TARGET RECON + PORT SCAN
Full enumeration of a single target: resolves MAC address, hardware vendor from a local OUI table, and hostname via reverse DNS and NetBIOS. Follows up with a fast nmap -F -T4 port scan and displays all open services. Everything written to the session loot file.
Vendor lookup: local OUI table — no network required.
All results logged to JSON loot file.
MODULE C
ROLE: OPSEC
MAC SPOOFING
Changes the interface MAC address before launching any attack so your real hardware identifier never appears in target ARP caches, switch logs, or captured traffic. Supports random MAC generation (with the locally administered bit set correctly) or manual entry of a specific MAC address.
Run before any attack for operational security.
Uses ip link — no external tools required.

// blue team

How to Protect Yourself

ARP attacks are possible because the protocol has no built-in authentication. KTOx includes 7 defensive modules alongside the attack tools — because understanding how to defend is just as important as understanding how to attack. Layer these defenses — no single measure is sufficient on its own.

👁
ARP WATCH — PASSIVE MONITOR
Passively sniffs all ARP traffic and builds a live IP→MAC mapping table. Fires an alert whenever an IP is claimed by a different MAC than previously recorded. Use this to detect if someone is already poisoning your network. All conflicts logged to the session file.
KTOx module [D] — sends no packets
📊
LIVE ARP DIFF — TABLE POLLING
Snapshots the OS ARP table on a configurable timer and diffs each iteration. More reliable than passive sniffing alone — catches poisoning even during silent ARP periods when no new packets are being sent. Alerts on changes, new entries, and removed entries.
KTOx module [F] — configurable poll interval
🚨
ROGUE DEVICE DETECTOR
Periodically re-scans the network and compares against the baseline MAC list. Fires an alert immediately when any device joins the network with a MAC address that wasn't present when the baseline was captured. Catches unauthorized devices and new attack tools connecting to the LAN.
KTOx module [G] — configurable scan interval
🔒
ARP HARDENING — STATIC ENTRIES
Scans the network, generates static ARP entries for every discovered host, and applies them to the OS with one command. Static entries cannot be overwritten by ARP replies — making poisoning impossible for hardened hosts. Also exports a shell script for re-application after reboot.
KTOx module [H] — auto-applies + saves arp_harden.sh
📋
NETWORK BASELINE EXPORT
Exports a full JSON snapshot of the current network state — IP, MAC, vendor, and hostname for every discovered host. Use as a trusted reference for ARP Diff comparisons and Rogue Device Detection. Stored in the loot directory with a timestamp for versioning.
KTOx module [I] — exports baseline_TIMESTAMP.json
🛡
DYNAMIC ARP INSPECTION (DAI)
Enterprise switches support DAI — they validate ARP packets against a DHCP snooping binding table and silently drop any that don't match. Stops ARP spoofing at the switch level before it reaches hosts. The most effective network-level defense for managed environments.
# Cisco IOS
ip arp inspection vlan 1
ip dhcp snooping vlan 1
📡
VPN ON UNTRUSTED NETWORKS
On networks you don't control, assume ARP poisoning is possible. A VPN tunnels all traffic through an encrypted channel — even if an attacker intercepts your traffic via MITM, they see only encrypted ciphertext. WireGuard is the recommended modern option.
sudo wg-quick up wg0
🧱
VLAN SEGMENTATION
ARP attacks are limited to the local broadcast domain. VLANs segment the network so an attacker on one VLAN cannot reach hosts on another. Isolate IoT devices, guest networks, and critical infrastructure into separate VLANs — ARP cannot cross VLAN boundaries without a router.
Separate VLANs for IoT / Guest / Infrastructure
